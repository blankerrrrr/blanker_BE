> 
> TASK 완료 후 커밋, 완료한 TASK에 체크할 것
> 
> 위에 있을수록 우선순위 높음

## 지시사항
- 모든 문서 작업은 기본적으로 한국어로 수행한다.
- 시키지 않은 짓은 하지 말 것

## 커밋 규칙
- `feat: 한국어 메시지` 형식
- 기능 하나 개발 후 커밋
- 커밋 후 변경사항 요약 문서 작성

### command
- **!!매우 중요!!** 절대 복합 명령어(&가 포함된 여러 개의 명령어)를 사용하지 않는다.
### allowed to commend
- git status *
- git diff *
- git add *
- git commit *
- git log *
  이 외에 단순 파일 읽기/테스트 등 코드베이스 및 운영체제에 영향을 끼치지 않는 명령어는 모두 허용함.

### restricted to commend
- git clone *
- git pull *
- git push *
- git checkout *
- rm *

허락없지 파일을 삭제해선 안되며, 지정된 범위 외의 작업 공간의 파일을 수정해선 안됨.

필요할 경우 요청할 것

## 출력 요청
출력은 결과만 간단하게 표시한다. 아래 포맷 중 하나를 골라서 출력한다.

### 1. 변경 사항 요약
> TASK 완료 후 변경 사항을 요악하여 출력합니다.

- 변경 파일을 나열한다.
- 변경 이유를 설명한다. 단 256자 이내로 제한한다.
- 변경된 플로우를 설명한다. (필요 시)
- 추가로 설명할 게 있다면 설명한다. (필요 시)

### 2. 파일 찾기 질문
> "어떤 문자열이 이 파일 내에 있어?" 같은 질문

찾는 것에 대한 파일 또는 폴더 위치를 제공한다.

출력 내용:
- 파일 위치: `systems/identity/identity-domain/...`
- 상세 위치: 117~123번째 줄

### 3. 구현 상태(플로우) 질문
> "어떤 요청을 보냈을 때 토큰이 어디에 저장돼?" 같은 질문

코드를 읽어 구현을 확인하고, 흐름에 대해 설명한다.

출력 내용:
- 확인한 파일: `systems/identity/identity-domain/...`, `systems/identity/identity-domain/...` (`,`로 구분)
- 플로우 설명:

### 4. 방향성 제시
> 구현 방향이나 수정 방향에 대한 질문이 들어왔을 경우

넘버 라벨링을 통해 구현 방향을 제시한다. 사유도 함께 작성한다.

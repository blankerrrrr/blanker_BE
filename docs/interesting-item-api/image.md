## 영화/드라마 — TMDB

포스터, 배경(backdrop), 스틸컷, 로고 등을 여러 사이즈로 줍니다. 다만 응답에는 경로만 담겨 오고, 앞에 이미지 베이스 URL을 붙여야 실제 이미지가 됩니다.

```
# 응답에 이런 필드가 옴
"poster_path": "/abc123.jpg"

# 실제 이미지 URL 조합
https://image.tmdb.org/t/p/{size}/{poster_path}
예: https://image.tmdb.org/t/p/w500/abc123.jpg
```

size는 `w200`, `w500`, `original` 등이 있습니다.

> 영화 이미지는 포스터를 사용함

---
## 애니 — Jikan (이미지 있음)

`images` 필드에 JPG/WebP 포스터가 여러 사이즈로 완전한 URL로 옵니다. TMDB와 달리 조합 없이 바로 쓸 수 있어요.

```
"images": { "jpg": { "image_url": "https://cdn.myanimelist.net/images/anime/..." } }
```
---
## 게임 — RAWG (이미지 있음)

게임 배경 이미지(`background_image`)와 스크린샷을 완전한 URL로 제공합니다. 스크린샷은 별도 엔드포인트로도 여러 장 받을 수 있어요.

```
"background_image": "https://media.rawg.io/media/games/..."
GET /games/{id}/screenshots?key={KEY}   # 스크린샷 여러 장
```
---
## 소설 — 알라딘 / 네이버 책 (표지 이미지 있음)

둘 다 책 표지 이미지 URL을 제공합니다. 알라딘은 `cover`, 네이버 책은 `image` 필드로 커버 이미지가 옵니다.

## 뮤지컬 — KOPIS (포스터 있음)
공연 포스터 이미지 URL을 제공합니다. 상세 조회 응답에 `poster` 필드로 포스터 이미지 경로가 옵니다.

## 웹툰 — korea-webtoon-api (썸네일 있음)

웹툰 썸네일 이미지 URL을 제공합니다. 다만 서드파티라 이미지 URL이 원본 플랫폼(네이버/카카오)을 가리키는 경우가 많아서, **핫링크 제한이나 CORS 문제**가 생길 수 있습니다. 실서비스라면 이미지를 자체 서버로 프록시하거나 캐싱하는 걸 고려해야 해요.

## 정리 표

| 카테고리 | 이미지 | 형태 | 주의 |
|---|---|---|---|
| 영화/드라마 (TMDB) | 포스터·배경·스틸 | 경로만 (베이스 URL 조합 필요) | size 지정 |
| 애니 (Jikan) | 포스터 | 완전한 URL | 바로 사용 |
| 게임 (RAWG) | 배경·스크린샷 | 완전한 URL | — |

## 실무 팁

이미지 처리에서 두 가지를 신경 쓰면 좋습니다.

**URL 형태가 API마다 다릅니다.** TMDB처럼 경로만 주는 곳은 베이스 URL을 붙여야 하고, 나머지는 완전한 URL이라 바로 씁니다. service 계층에서 이미지 URL을 완성된 형태로 통일해서 응답하면 프론트가 편해집니다.

**외부 이미지 직접 노출의 리스크.** 특히 웹툰처럼 원본 플랫폼 이미지를 그대로 쓰면, 그쪽에서 핫링크를 막거나 URL이 바뀌면 이미지가 깨집니다. 트래픽이 커지면 이미지를 본인 스토리지(S3 등)에 캐싱하거나 프록시하는 구조를 고려하세요. TMDB·RAWG처럼 CDN을 제대로 갖춘 곳은 직접 참조해도 대체로 안정적입니다.

정리하면, **웹툰만 이미지 처리에 약간 주의가 필요하고 나머지는 다 이미지를 잘 제공합니다.** 특정 API의 이미지 필드를 Pydantic 스키마로 매핑하는 구체적 코드가 필요하면 알려주세요.
